Thesis Running document and thought process for all thesis work.

Problem statement
Is there a process to keep the natural flow of discussion continuous while also providing useful & relevant data based on the discussion context?

Live conversations are fluid and natural. A conversation makes agreements, takes unpredictable turns, and could even come to arguments. Assume a user engaged in a live conversation often needs information to validate a point or argument. Traditionally, a user is forced to pause this conversation and go retrieve information which is useful and relevant to the context of the conversation. What if we could minimize this problem to where no pause occurs in the conversation, but instead continues this fluid motion?

For example, a presenter of information tends to promote discussion throughout a presentation. In particular, the discussion surrounding the topic can lead to problems when the talking points deviate from the intended ordered design of the presentation. The audience, could potentially focus their attention on irrelevant presentation data, or even be forced to remove their attention and query the information themselves.

A process or software tool in which injects reliable information based on the surrounding discussion or topic has a possibility of enhancing a learning process and continuing the ongoing discussion. In order to solve this problem, its been broken down into three question segments:

First, can we store unstructured document data in an organized manner so it can later be retrieved? Information comes in many forms like web pages, slides, pdf, word documents. Storing all of these documents into a single database instance for searching will require an organized relational design. The extraction is another key focus point in this problem. Each document type will require a proprietary extraction method to properly store itâ€™s containing information.

Next, can we query these stored unstructured documents based on a surrounding discussion or topic? In other words, can we use a text based discussion which is currently ongoing to query a database of possible relevant document data?

Lastly, can we inject these unstructured documents into a platform in which the end user can view and use? In theory, this would allow the user to view the most relevant and useful information based on the surrounding discussion at the given moment.

Repetitive Searching over Active Conversation

*This Validation piece is based on a active presentation. After the pivot to document injection, this needs revised and modified.

The validation will hold true if the results produced by the active queries, based on a timing parameter, are relevant to the data collected a time t. If the time is broken down, it can be shown at which moments in time can be validated with the results:

Let t be real time seconds, d be keywords/phrases in which to use as a parameter for queries, r be results from query, and P be presentation data from database.

The query is executed on the data model at t1 using d0 as it’s parameter condition. d0 is created using the mining method on the text data gathered at the times t0 → t1. The results of r1 will be produced at the end of the t0 → t1 at time t1. Then r1 must be validated.

The validation of r1 will start by evaluating the count of the query. This process will finish at time t1L. The tuple count will determine which path to route the system.

If count(r1) = 0, meaning d0 was insufficient in carrying relevant results in comparison to the provided P, the system will simply do nothing. By trusting our mining method and assuming the conversation text is accurate, the system will avoid displaying potentially irrelevant presentation data. The system will move on, and use d1 which is a result from times t1 → t2 and execute a new query.

If count(r1) = 1, the result is used as the selection.

If count(r1) > 1, there will need to be a selection process. This could involve setting priority on some fields and using an order by to grab the top result. The selection process can be interchangeable as long as we narrow to a single result.

Once results are validated and a selection is made, P1 is displayed. There will be a delay in time before P1 is displayed due to the validation of r1. This time is marked at t1L which is significant to note because it falls on the t1→ t2 timeline. This will cause each time segment to have its own thread. A new thread will be created at the start of each t interval.

After repeated iterations of the above process, we will need to use the count(r) to validate the results. We can use the basic methods precision and recall to quantify how the query selection is performing. Recall gives a useable result because we can use the query tuples returned over the total pieces of presentation data available.

Recall = (count(r) / total d))
